# Sambo (Phaser) — MIDI-driven scrubbing playback (technical spec for Codex)

## Scope

Implement MIDI-based music playback where the **player position along X** controls a **playhead**. When the player stops, playback stops; moving backward scrubs the playhead backward. Level authoring is in **musical ticks/beats**, not seconds. Audio is generated by a **custom WebAudio synth** (not pre-rendered audio). Program Change is ignored.

This spec covers:

* MIDI concepts required (ticks, delta-time, tempo map, NoteOn/Off semantics)
* Data model and preprocessing
* Runtime playhead update logic (forward + reverse)
* Handling tempo changes and difficulty speed scaling
* Polyphony/stuck-note robustness for scrubbing

---

## MIDI facts/assumptions (load-bearing)

1. **Standard MIDI File (SMF)** stores event timing as **delta-time**, not absolute time. Absolute time is obtained by summing deltas. ([Stack Overflow][1])
2. Delta-times are encoded as **VLQ (variable length quantity)** primarily for space efficiency. ([midi.teragonaudio.com][2])
3. Tempo is defined by **Set Tempo meta event (0xFF 0x51)** carrying **microseconds per quarter note (µs/QN)**. BPM is `60,000,000 / usPerQuarter`. ([basicsynth.com][3])
4. **Note On with velocity 0** must be treated as **Note Off**. ([MIDI.org][4])

---

## Design goals

* Editor axis is **tick-based**: objects/markers aligned to beats/ticks.
* Runtime playhead is derived from player X: `playheadTick = kTickPerUnit * (x - x0)` (linear in X).
* Audio playback follows MIDI content **including tempo changes**.
* Difficulty modifies overall musical speed via a **relative multiplier** (e.g., hard = 1.25×) while preserving tempo-change shape.
* Robust scrubbing: no stuck notes, deterministic state, stable under large jumps/teleports.

---

## Data model

### Parsed MIDI (preprocess output)

* `ppq: number` (ticks per quarter note; from MIDI header)
* `tempoPoints: TempoPoint[]` sorted by ascending tick

  * `TempoPoint = { tick: int, usPerQuarter: int }`
  * If missing, assume 500000 µs/QN (120 BPM) per SMF convention (optional default behavior).
* `notes: NoteInterval[]` sorted by `startTick` (and optionally also by `endTick`)

  * `NoteInterval = { startTick: int, endTick: int, pitch: int, velocity: int, trackId: int, channel: int }`

### Voice manager (runtime)

Since overlapping identical pitches are legal, manage polyphony with **counts** per key:

* `key = trackId + ":" + channel + ":" + pitch`
* `voiceState[key] = { count: int, voice: SynthVoice }`

`SynthVoice` is your WebAudio voice: oscillator(s) + gain envelope, etc.

## Preprocessing pipeline (MIDI → tick intervals)

### Step 1: Parse tracks into absolute tick events

For each track:

* Initialize `absTick = 0`
* For each event:

  * `absTick += deltaTick` (delta-time is VLQ) ([midi.teragonaudio.com][2])
  * If event is Set Tempo meta (0xFF 0x51), append `tempoPoints.push({tick: absTick, usPerQuarter})` ([basicsynth.com][3])
  * If event is channel note message:

    * Normalize `NoteOn(velocity=0)` to `NoteOff` ([MIDI.org][4])
    * Emit a normalized `NoteOn` or `NoteOff` with:

      * `tick = absTick`, `pitch`, `velocity`, `channel`, `trackId`

### Step 2: Build note intervals using stacks per (track, channel, pitch)

Maintain:

* `open[key] = stack<int startTick>`

Rules:

* On `NoteOn` (velocity > 0):

  * `open[key].push(absTick)`
  * store velocity (either on stack as `{startTick, velocity}` or in a parallel stack)
* On `NoteOff`:

  * If `open[key]` not empty:

    * `startTick = open[key].pop()`
    * create `NoteInterval{startTick, endTick=absTick, pitch, velocity, trackId, channel}`
  * Else:

    * “orphan NoteOff”: ignore or log (file is dirty)

Notes:

* This supports overlapping same pitch (stack semantics).
* Program Change events are ignored by design.

### Step 3: Canonicalize outputs

* Sort `tempoPoints` by `tick`.
* Ensure at least one tempo point at tick 0:

  * If first tempo point is not at 0, insert default at 0 or treat as implicit.
* Sort `notes` by `startTick`. Also produce `notesByEnd` sorted by `endTick` (recommended for runtime).

## Tempo map usage (tick ↔ seconds)

### Core formula (segment-local)

For a tempo segment with `usPerQuarter` and `ppq`:

* `secPerTick = (usPerQuarter / 1e6) / ppq` ([basicsynth.com][3])

### Requirement: respect tempo changes

Implement conversion by summing across tempo segments:

* For any `tick T`, compute `sec(T)` by walking tempoPoints segments:

  * For each segment `i` from `tempoPoints[i].tick` to `tempoPoints[i+1].tick` (or ∞ for last):

    * `tickSpan = clamp(T, segStart, segEnd) - segStart`
    * `sec += tickSpan * secPerTick_i`
      Stop when `T` is within processed range.

### Difficulty scaling (relative speed)

Use a speed multiplier `speed` (normal=1.0; hard>1.0):

* `sec_scaled(T) = sec(T) / speed`
  This preserves the *shape* of tempo changes while increasing overall tempo proportionally.

## Runtime mapping: player X → playhead tick

Given linear mapping:

* `playheadTick = tickPerUnit * (x - x0)`
* Clamp to `[0, songEndTick]` as needed.

Maintain:

* `tickPrev`, `tickNow` each frame.

## Runtime note-state update (forward + reverse scrubbing)

### Key invariant

A note is active at tick `t` iff:

* `startTick <= t < endTick`

### Two modes: incremental vs rebuild

Define:

* `scrubThresholdTick` (e.g., equivalent of 0.25–0.5 s at typical tempo, or a fixed tick count)
  If `abs(tickNow - tickPrev) > scrubThresholdTick`, do **rebuild**; else do **incremental**.

#### Rebuild mode (robust for teleports/large deltas)

1. `panicAllNotesOff()`:

   * for each active `key`, set `count=0`, apply short release, stop/disconnect voice.
2. Determine active notes at `tickNow`:

   * simplest: iterate all notes and select `start<=tickNow<end` (ok for small MIDI)
   * preferred: interval index (e.g., bucket notes by bar/beat range) or sweep-line pointers
3. For each active note:

   * `noteOn(key, velocity, attackShort=true)` (attack short even if “mid-note”)
   * increment `count`

#### Incremental mode (small deltas)

Let `tA = min(tickPrev, tickNow)`, `tB = max(tickPrev, tickNow)`.

Maintain two sorted arrays:

* `notesByStart` sorted by `startTick`
* `notesByEnd` sorted by `endTick`

Update depends on direction:

**Forward (tickNow > tickPrev):**

* For each note with `startTick in (tickPrev, tickNow]`:

  * `noteOn(key, velocity, attackShort=true)` and `count++`
* For each note with `endTick in (tickPrev, tickNow]`:

  * `noteOff(key)` and `count--` (if reaches 0: release+stop)

**Reverse (tickNow < tickPrev):**

* For each note with `endTick in [tickNow, tickPrev)`:

  * note becomes active again when scrubbing back before its end:
  * `noteOn(key, velocity, attackShort=true)` and `count++`
* For each note with `startTick in [tickNow, tickPrev)`:

  * note becomes inactive when scrubbing back before its start:
  * `noteOff(key)` and `count--`

Implementation detail: use binary search to find index ranges in the sorted arrays and iterate only the slice.

## Synth/voice behavior constraints

* On `noteOn`: create or reuse `SynthVoice` for `key`.

  * Apply **short attack** (e.g., 5–20 ms) to avoid clicks when enabling mid-note.
  * Map velocity (0–127) to gain (linear or curve).
* On `noteOff`: apply short release (e.g., 20–80 ms) and stop oscillator after release when `count==0`.
* Because overlaps are legal, `count` must not go negative; clamp and log if it does.

---

## What “reverse” means in this design

This design provides **reverse scrubbing in the score/timeline sense**:

* As the playhead moves backward, the set of active notes is updated to match the earlier time.
  It does **not** produce “true reversed audio waveform” (that would require reversing rendered audio, not MIDI-driven synthesis).

---

## Edge cases / robustness

1. **Orphan NoteOff** (no matching NoteOn in stack): ignore/log.
2. **Unclosed NoteOn at end-of-track**: either drop or close at `songEndTick`.
3. **Tempo event ordering**: tempo is global in practice; if multiple tracks contain tempo events, merge and sort by tick.
4. **Simultaneous events** (delta=0): treat as same `absTick`—supports chords and simultaneous note boundaries.
5. **Channel separation**: matching NoteOff must be within the same channel as NoteOn (use key with channel).

## Minimal API surface for Codex implementation

### Preprocess module

* `parseMidiToTickModel(midiBytes) -> { ppq, tempoPoints, notesByStart, notesByEnd, songEndTick }`

### Runtime module

* Inputs per frame: `playerX`, `dt`, `difficultySpeed`
* State: `tickPrev`, `activeVoiceState`
* Functions:

  * `tickNow = mapXToTick(playerX)`
  * `updateActiveNotes(tickPrev, tickNow)` (incremental/rebuild)
  * `tickPrev = tickNow`

### Optional helpers

* `sec = tickToSec(tick, tempoPoints, ppq) / speed` (only if needed for UI/debug or scheduling)
* Binary search helpers for `[t1, t2]` range extraction in sorted arrays.

---

## Acceptance checks

* Stopping player stops audio immediately (no drift).
* Moving forward/backward scrubs deterministically; no stuck notes after repeated direction changes.
* Tempo changes in MIDI audibly affect spacing of events when scrubbing (because tick→sec conversion is tempo-aware, and note activity is tick-based).
* Difficulty speed multiplier uniformly speeds/slows playback while preserving tempo-change relative structure.
