**Status:** Draft
**Author:** (TBD)

## 1. Summary

This RFC specifies the implementation of a **custom WebGL background pipeline** in Phaser 3 that renders **12 harmonic bands** driven by real-time MIDI pitch-class activity.

The background reacts deterministically to Sambo’s tick-based MIDI scrubbing system and must remain visually minimal, performant, and non-intrusive.

This document defines requirements, constraints, interfaces, rendering semantics, and acceptance criteria.

No implementation is included.

---

## 2. Motivation

Sambo’s core mechanic links player motion to musical scrubbing.
A reactive visual background strengthens:

* Musical readability
* Player feedback
* World coherence

However, traditional spectrum visualizers are too noisy.
This system provides a **minimal harmonic field** aligned with Sambo’s aesthetic direction.

---

## 3. Scope

### In Scope

* Custom Phaser 3 WebGL pipeline
* Full-screen fragment shader
* 12 pitch-class driven horizontal bands
* Uniform-based runtime control
* Deterministic behavior under forward/reverse scrubbing

### Out of Scope

* MIDI parsing
* WebAudio synthesis
* Gameplay logic
* Post-processing effects
* Asset loading systems

---

## 4. System Architecture

### 4.1 Data Flow

```
MIDI Tick Engine
        ↓
Active Voice State
        ↓
Pitch-Class Bin Computation (JS)
        ↓
Uniform Update
        ↓
WebGL Shader
        ↓
Fullscreen Background Render
```

The shader must remain purely visual and stateless.

---

## 5. Musical Data Contract

The rendering system depends on the following runtime data.

### 5.1 Pitch-Class Bins

Type:

```
Float32Array(12)
```

Each index corresponds to:

```
pitchClass = pitch % 12
```

Range:

```
0.0 – 1.0
```

Definition:

* Derived from currently active notes.
* Weighted by velocity and/or voice count.
* Normalized.
* Smoothed (low-pass filter recommended).
* Must not allocate new arrays per frame.

---

### 5.2 Intensity

Scalar:

```
uIntensity ∈ [0, 1]
```

Derived from musical density and/or player mechanic.

Controls:

* Global brightness
* Band thickness
* Visual contrast

---

### 5.3 Beat Phase

Scalar:

```
uBeatPhase ∈ [0, 1]
```

Represents fractional beat progression.

Used only for subtle rhythmic animation.

---

### 5.4 Time

Scalar:

```
uTime (seconds)
```

Continuous increasing value.

Used for lightweight oscillation.

---

### 5.5 Resolution

Vector:

```
uResolution = vec2(width, height)
```

Used for coordinate normalization.

---

## 6. Rendering Semantics

### 6.1 Layout

The vertical axis is divided into 12 equal regions.

Band centers:

```
y_i = (i + 0.5) / 12.0
```

Each pitch-class bin drives one band.

---

### 6.2 Band Profile

Each band uses a smooth falloff function:

Preferred: Gaussian profile

Parameters:

* Base thickness scales with `uIntensity`
* Energy scales with `uPC[i]`

Band amplitude must be continuous and smooth.

---

### 6.3 Color Policy

Colors must:

* Remain low-saturation
* Stay consistent with Sambo’s dark lunar palette
* Avoid harsh RGB cycling
* Avoid bright neon or arcade appearance

Subtle per-band hue variation is allowed.

---

### 6.4 Animation Policy

Allowed:

* Subtle breathing effect linked to `uBeatPhase`
* Light oscillation via `uTime`

Forbidden:

* Hard flashes
* Rapid flicker
* Discontinuous jumps
* Per-pixel noise

---

### 6.5 Background Stability

When:

* All bins = 0
* Intensity = 0

The background must render near-dark with minimal residual glow.

---

## 7. Phaser 3 Integration Requirements

### 7.1 Pipeline Type

Must extend:

```
Phaser.Renderer.WebGL.Pipelines.SinglePipeline
```

Must:

* Load custom fragment shader
* Accept uniform updates
* Render full-screen

---

### 7.2 Uniform Interface

The pipeline must expose methods to set:

| Uniform     | Type        |
| ----------- | ----------- |
| uResolution | vec2        |
| uTime       | float       |
| uIntensity  | float       |
| uBeatPhase  | float       |
| uPC[12]     | float array |

Uniforms must be clamped before use.

---

### 7.3 Activation

The pipeline must:

* Be registered in Phaser’s pipeline manager
* Be attached either to:

  * The main camera
  * Or a fullscreen background object

It must render behind gameplay layers.

---

## 8. Performance Requirements

* Shader loop size fixed to 12 iterations
* No dynamic loops
* No texture sampling
* No branching based on runtime values
* No per-frame allocations
* Stable at 60 FPS
* < 10 ms render cost on mid-tier hardware

---

## 9. Robustness Requirements

The system must behave correctly when:

* No active notes
* Rapid scrubbing forward/backward
* Teleport rebuild mode
* Resolution changes
* Intensity edge values (0 or 1)

No visual artifacts may persist after scrubbing.

---

## 10. Acceptance Criteria

The implementation is accepted when:

✔ Compiles in Phaser 3.x
✔ Renders 12 harmonic bands
✔ Reacts deterministically to MIDI scrubbing
✔ Shows no stuck glow artifacts
✔ Preserves gameplay readability
✔ Scales correctly with resolution
✔ Meets performance targets

## 11. Risk Analysis

### Risk: Over-visualization

Mitigation: clamp intensity, enforce low color saturation.

### Risk: Performance regression

Mitigation: fixed iteration count, no texture usage.

### Risk: Visual jitter

Mitigation: smoothing in JS before uniform upload.

# Annex A — Visual Style (Exact Color Constants)

**Applies to:** RFC-001 Harmonic Bands WebGL Background Pipeline (Phaser 3)
**Goal:** Provide a strictly-defined palette and numeric parameters so Codex can implement the shader consistently and gameplay-safely.

---

## A1. Design Constraints

1. **Background-first:** Must not compete with foreground platforms, player, hazards, UI.
2. **Low saturation, low contrast:** The effect should read as atmospheric illumination.
3. **Intensity-gated:** Visual energy must scale primarily with `uIntensity` and secondarily with `uPC[i]`.
4. **No noise textures:** Avoid film grain / per-pixel randomness.
5. **Consistency:** Same MIDI state → same visual output (deterministic).

---

## A2. Master Palette (Sambo “Deep Lunar”)

These are the only base colors permitted in the shader unless explicitly extended.

### Core background

* **BG0 (Deep Space):** `#05070F`  (RGB 5, 7, 15)
* **BG1 (Deep Slate):** `#0B1020`  (RGB 11, 16, 32)
* **BG2 (Slate Ink):** `#111A2E`   (RGB 17, 26, 46)

### Neutral glow anchors

* **GlowCool (Steel Blue):** `#3A4663` (RGB 58, 70, 99)
* **GlowWarm (Ochre Gold):** `#C9A227` (RGB 201, 162, 39)

### Safety highlight

* **MoonIvory (Very low use):** `#E8E6E3` (RGB 232, 230, 227)
  Constraint: must never exceed 5% contribution to final color per pixel.

---

## A3. Harmonic Band Palette (12 pitch classes)

Define a **12-entry band color table** `BandRGB[12]`.
Each entry is an RGB hex constant.

This table is designed to be subdued, alternating cool/warm hues while staying within the “Deep Lunar” theme.

|  i | Pitch Class | Color            | Hex       |
| -: | ----------- | ---------------- | --------- |
|  0 | C           | Deep Steel       | `#2B344C` |
|  1 | C#          | Steel            | `#313B55` |
|  2 | D           | Slate            | `#374261` |
|  3 | D#          | Blue Slate       | `#3B4B6B` |
|  4 | E           | Cool Teal Slate  | `#345663` |
|  5 | F           | Teal Ink         | `#2F4D57` |
|  6 | F#          | Warm Slate       | `#4A4A55` |
|  7 | G           | Warm Brown Slate | `#5A4D4B` |
|  8 | G#          | Brown Ochre      | `#6B5542` |
|  9 | A           | Ochre            | `#7A5E3A` |
| 10 | A#          | Muted Gold       | `#8A6A33` |
| 11 | B           | Pale Gold Slate  | `#6B5F3A` |

**Constraint:** These are *base band colors*. Final output must still be dominated by BG0/BG1.

## A4. Brightness and Contrast Limits

All final outputs must obey these limits to remain gameplay-safe.

### Luma caps (approximate)

* **At `uIntensity = 0`:** average luma should remain near BG0/BG1 (almost black)
* **At `uIntensity = 1`:** peak luma should still be capped (no full whites)

Practical constraints Codex must enforce:

* Clamp any per-band additive contribution to `≤ 0.25` in linear RGB magnitude (before final clamp).
* Ensure final color is clamped to `[0,1]` as normal, but avoid reaching near-1 values except extremely rarely.

---

## A5. Alpha / Transparency Policy

This background is **opaque** (`alpha = 1.0`) and rendered behind gameplay.
If you need blending with other background layers, handle it at the scene compositing level, not inside the shader.

---

## A6. Shader Numeric Constants

Codex must implement these constants to standardize “feel”.

### Band geometry

* `N_BANDS = 12`
* Band center:

  * `y_i = (i + 0.5) / 12.0`

### Thickness (Gaussian sigma in normalized UV space)

* `sigmaMin = 0.010`
* `sigmaMax = 0.022`
* `sigma = mix(sigmaMin, sigmaMax, uIntensity) * (0.70 + 0.60 * uPC[i])`

### Amplitude (band strength)

* `ampBase = 0.015`
* `ampMax = 0.180`
* `amp = (ampBase + ampMax * uPC[i] * uIntensity)`

### Glow shaping

* Use a soft exponent for glow:

  * `glow = pow(gauss, 0.65)`
    where
  * `gauss = exp(-(dy*dy)/(2*sigma*sigma))`

### Beat breathing (very subtle)

* `breatheAmp = 0.08` (max ±8%)
* `breathe = 1.0 + breatheAmp * sin(2π * (uBeatPhase + 0.15*sin(0.5*uTime)))`

### Vignette

* Apply only mild vignette:

  * `vStrength = 0.15`
  * `v = 1.0 - vStrength * ((uv.x-0.5)^2 + (uv.y-0.5)^2)`
  * Multiply final color by `v`

---

## A7. Intensity and Bin Smoothing Requirements (JS-side)

To prevent jitter:

### Pitch-class smoothing

Use exponential smoothing per bin:

* `tauPC = 0.20 s` (recommended)
* `alpha = 1 - exp(-dt / tauPC)`
* `pcSmooth[i] = lerp(pcSmooth[i], pcRaw[i], alpha)`

### Intensity smoothing

* `tauIntensity = 0.12 s`

### BeatPhase

No smoothing required (should be computed analytically).

---

## A8. “Do Not Break” Rules

Codex must ensure:

1. **No hard outlines** on bands.
2. **No high-frequency noise**.
3. **No rapid flicker** (bin smoothing mandatory).
4. **No overbright** results (respect amplitude caps and MoonIvory constraint).
5. **No color cycling** beyond the fixed band palette.

---

## A9. Reference Conversion

Codex must convert hex colors to normalized linear floats (or sRGB floats consistently).
If using sRGB directly in shader without linearization, keep all colors and calculations consistent in that space.

Recommended: keep it simple and use sRGB floats consistently, because the palette is designed to be safe under that assumption.
